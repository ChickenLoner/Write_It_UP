# Write_It_UP

Personal CTF / cybersecurity lab write-up collection. Markdown write-ups are
built into a SOC-themed static site by `build_md_to_html_with_toc.py` and
deployed to GitHub Pages (see `.github/workflows/deploy.yml`).

## When adding a new write-up

After adding or renaming a write-up markdown file, **add a matching entry to
`writeups_meta.json`** — this drives the index card (difficulty pill, category,
summary, tags). Without it the card still renders, but shows the name only.

The JSON is keyed by the md file's path relative to the repo root. Add:

```json
"<Folder>/<[Platform Write-up] Name>.md": {
  "difficulty": "Very Easy | Easy | Medium | Hard | Insane | Unknown",
  "category": "1-3 word Title-Case domain, e.g. Network Forensics / AD / Kerberos",
  "tags": ["3-6", "short", "lowercase", "tools-or-techniques"],
  "summary": "One active-voice sentence <=155 chars; never start with \"This lab\""
}
```

Derive the values by reading the write-up's scenario + questions. Labelling
rules and the bulk regeneration pipeline are documented in `tools/README.md`.

Keep the entry a single lab — no need to re-run the batch generator for one file.

## Build

`build/` is generated by CI and git-ignored — never commit it. To build locally
on Windows run with `PYTHONUTF8=1` (the script prints ✅, which crashes the
cp874 console otherwise).

What the build does and does not publish:

- Write-ups only. Markdown at the repo root (`README.md`, `CLAUDE.md`) and repo
  docs anywhere (`SKIP_NAMES` in the build script) are never rendered; `tools/`
  and `re-design/` are not walked at all.
- Shared CSS is written once to `build/assets/{base,article,index}.css` with a
  content-hash query string, not inlined per page.
- `build/resources/` gets only the images the generated HTML actually
  references, re-encoded to **lossless** WebP (~58% of the PNG bytes, no
  quality loss — readers zoom into hex dumps, so lossy is not acceptable).
  Originals are kept whenever WebP would be larger. `<img>` tags are rewritten
  in a post-pass and stamped with intrinsic `width`/`height` to stop layout
  shift. `resources/` on disk is ~1 GB and GitHub Pages caps a published site
  at 1 GB, so unreferenced files are dropped too.
- Encoding is cached in `.imgcache/` (git-ignored, restored by `actions/cache`
  in CI). Resource filenames are content hashes, so a cached WebP never goes
  stale — only newly added screenshots cost encode time. Set
  `SKIP_IMAGE_OPT=1` to skip encoding entirely for a fast local build.
- The build prints how many referenced images are **missing** from
  `resources/`. Those render broken. Files lost in the `_resources/` →
  `resources/` rename are usually still in git history:
  `git rev-list --all --objects | grep <name>` then `git cat-file -p <sha> > resources/<name>`.
- `sitemap.xml`, `robots.txt` and `_headers` are generated from the write-up
  list. `_headers` is Cloudflare-specific and inert on GitHub Pages.

## Hosting

Live at <https://writeups.chicken0248.fyi> on GitHub Pages (`deploy.yml`, push
to `main`). Custom domain pinned by the root `CNAME`.

Capacity, as of the WebP change: **0.495 GiB across 7,868 files**, averaging
~1.4 MiB and ~21 images per write-up.

- GitHub Pages caps a published site at **1 GB, and that is a hard limit** —
  before the WebP pass the site was at ~0.96 GiB, roughly 40 MB from failing to
  deploy. Current headroom is ~368 more write-ups. Bandwidth (100 GB/mo) is the
  soft limit.
- `cloudflare-preview.yml` (manual `workflow_dispatch` only) builds the same
  tree and Direct-Uploads it to Cloudflare Pages for migration testing on
  `*.pages.dev`. It does not affect the live site.

If capacity ever becomes the problem, Cloudflare Pages free has no documented
total-size cap; the binding limit is **20,000 files per deployment** (~583 more
write-ups) and 25 MiB per file, with static-asset requests free and unlimited.
Two things to know before committing to it: Cloudflare caps builds at 20
minutes on every plan, which the cold WebP encode exceeds — hence Direct Upload
from GitHub Actions rather than Cloudflare's git integration — and a Direct
Upload project can never be converted to git integration afterwards.
